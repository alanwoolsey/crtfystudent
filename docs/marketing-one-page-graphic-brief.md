# crtfy student One-Page Marketing Graphic Brief

Use this document as the full prompt/context for generating a polished one-page marketing graphic for `crtfy student`.

The goal is not a UI screenshot collage. The goal is a high-end visual story that explains the product clearly and makes the category shift feel obvious.

---

## 1. What the product is

`crtfy student` is an admissions operating system for small schools.

It is built for admissions employees who spend their day moving students from:

`Inquiry -> Applicant -> Incomplete -> Complete -> Review -> Decision -> Admit -> Deposit -> Enrolled`

The product is not trying to rebuild Slate.

It is not trying to become a generic CRM.

It is not trying to replace Element451 as a chatbot / campaign engine.

It is designed to be better for the admissions operator who needs to:

- know who to work right now
- reduce incomplete applications
- handle transcripts and documents correctly
- make confident decisions
- move admits to deposit
- reduce melt

---

## 2. Product position in the market

The market framing is:

- Slate = process and workflow engine
- Element451 = engagement and conversion automation
- `crtfy student` = admissions operations + transcript intelligence + trust + decision intelligence

The most important wedge:

`We help schools get applications completed faster, decisions made faster, and deposits protected better.`

The strongest differentiated themes are:

- incomplete-to-complete execution
- transcript evaluation
- document trust and provenance
- transfer certainty
- explainable decision packets
- daily work queues for admissions staff

---

## 4. What the image should communicate

The image should make these ideas instantly clear:

### Core message

## 5. Visual story to tell

The best one-page graphic should show a left-to-right or top-to-bottom operating story, not a random set of features.

Recommended story structure:

### Section 1: The admissions funnel

Show the admissions journey:

`Inquiry -> Applicant -> Incomplete -> Complete -> Review -> Decision -> Admit -> Deposit -> Enrolled`

But emphasize where `crtfy student` creates leverage:

- Incomplete
- Review
- Decision
- Yield / Deposit

### Section 2: The operator workspace

Show that the operator works from:

- Today’s Work
- Student 360
- Decision Studio
- Trust Center

These should read like a connected operating system, not isolated products.

### Section 3: The differentiated engine underneath

Show the hidden logic layer that makes the product different:

- transcript ingestion

- checklist progress

- explainable recommendation
- handoff and downstream readiness

### Section 4: The outcome

Show the practical results:

- fewer stalled applications
- faster decisions
- cleaner document operations
- better deposit conversion
- lower melt
